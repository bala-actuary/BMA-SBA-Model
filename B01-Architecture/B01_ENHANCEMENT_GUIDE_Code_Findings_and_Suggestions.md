**The critical detail B01 is missing**:
- Phase 1 runs once (cached in `initialised_settings`)
- Phases 2-3 run once (cached in `initialised_model`)
- Phase 4 runs **many times** (iterative liabilities + assets for goal-seeking)
- Phase 5 runs once at the end

This iteration is where the **SBA optimization** happens (not mentioned in current B01).

**Suggested Enhanced Section**:
```markdown
### Section Y: The 5-Phase Execution Pipeline

The run-client command executes in phases, each with distinct caching behavior:

**Phase 1: Settings Loading** (one-time, cached)
- File: main_client.py lines 72-80
- Loads Excel configuration: setup.xlsm → Settings object
- Cache flag: initialised_settings prevents reload on retry

**Phase 2: Model Initialization** (one-time, cached)  
- File: main_client.py lines 81-90
- Creates instrument objects, market projections, balancesheet
- Cache flag: initialised_model prevents reload on goal-seek loops
- Functions: initialise_model(), initialise_asset(), MarketProjection()

**Phase 3: Model Calibration** (one-time, follows Phase 2)
- File: main_client.py line 91
- Calibrates z-spreads, interest rate models, and rebalancing parameters
- Function: calibrate_model()

**Phase 4: Valuation (Iterative)** (runs many times during goal-seek)
- **Phase 4A**: Liability valuation via project_liability()
  - Discounts liability cashflows using calibrated interest rate curves
- **Phase 4B**: Asset valuation via project_assets()
  - Discounts asset cashflows, applies rebalancing logic, calculates SBA BEL
- These phases run in loops as the goal-seeker adjusts asset/liability proportions

**Phase 5: Results Writing** (one-time, final)
- File: main_client.py lines 95-100
- Writes Summary DataFrame to Excel via writer_excel.py
- Function: write_results()

**Why This Matters**: The caching behavior ensures model initialization overhead only 
occurs once, while the optimization loop (Phases 4A/4B) repeats efficiently.
```

### Finding 3: The 8-Step Module Initialization Order

**What the current B01 says**:
> "The loaders create instruments, markets provides rates, projection runs the calculation."

**What the code actually shows** (from `initialise_model()` in main_client.py):

When the model initializes, these objects are created in this **specific order**:

1. **SettingsLoader** retrieves all Excel parameters
2. **AssetParameterProvider** loads asset lookup tables
3. **Asset Factory** creates 20+ instrument objects (Bond, Swap, Equity, etc.)
4. **MarketProjection** creates base + stressed spot curves (90 curves total)
5. **BalanceSheet** object initialized with assets + liabilities
6. **AssetProjector** component ready to project assets forward
7. **RebalancingStrategy + SwapRebalancer** initialized for hedging
8. **BalanceSheetProjector → SBAGoalSeeker** instantiated for optimization

**Why order matters**:
- You can't create Assets (step 3) before loading parameters (step 2)
- You can't project balance sheet (step 8) before all components exist (steps 1-7)
- This is implicit in current code but **not explained** in B01

**Suggested Enhanced Section**:
```markdown
### Section Z: 8-Step Module Initialization Sequence

When initialise_model() is called (main_client.py line ~82), objects are 
created in this specific order (dependency graph):

**Step 1: Settings Loader**
- File: alm/model/loaders/settings_loader.py
- Class: SettingsLoader
- Action: Parse setup.xlsm → Settings dataclass with all parameters
- Output: settings object containing ~50 parameters

**Step 2: Asset Parameter Provider**
- File: alm/model/instruments/asset_parameter_provider.py
- Class: AssetParameterProvider
- Action: Load asset lookup tables from Excel
- Dependency: Requires settings.asset_lookup_file

**Step 3: Asset Factory (initialization_asset)**
- File: alm/model/instruments/asset_generation.py
- Function: initialise_asset() × N
- Action: Create Bond, Swap, Equity, Liability objects from portfolio CSV
- Dependency: Requires AssetParameterProvider + settings

**Step 4: Market Projection**
- File: alm/model/markets/market_projection.py
- Class: MarketProjection
- Action: Generate 90 spot curves (Base + 8 stress scenarios, 10 years forward)
- Dependency: Requires settings (scenario definitions)

**Step 5: Balance Sheet**
- File: alm/model/projection/balance_sheet_projection.py
- Class: BalanceSheet
- Action: Container holding assets + liabilities + market rates
- Dependency: Requires Assets (step 3) + Markets (step 4)

**Step 6: Asset Projector**
- File: alm/model/projection/asset_projector.py
- Class: AssetProjector
- Action: Ready to project assets forward (used in valuation loop)
- Dependency: Requires BalanceSheet (step 5)

**Step 7: Rebalancing Strategy**
- Files: alm/model/projection/alm_rebalancing.py + swap_rebalancer.py
- Classes: RebalancingStrategy, SwapRebalancer
- Action: Initialize hedging parameters and swap catalog
- Dependency: Requires Settings + Assets (step 3)

**Step 8: Balance Sheet Projector → SBA Goal Seeker**
- Files: alm/model/projection/balance_sheet_projector.py, sba_goal_seeker.py
- Classes: BalanceSheetProjector, SBAGoalSeeker
- Action: Ready to run optimization loop
- Dependency: Requires all of steps 1-7

This order is enforced implicitly by Python imports and function execution. 
Skipping or reordering steps would cause runtime errors.
```
